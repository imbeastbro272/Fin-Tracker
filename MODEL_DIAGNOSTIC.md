# Model Diagnostic - Why DT Predicts High Brightness in Bright Light

## 🐛 The Problem You Found

**Observation:**
```
Lux > 2000 + Motion detected → DT predicts 70% brightness ❌
Expected: Should predict LOW brightness (0-20%)
```

**Why this is wrong:**
When it's already bright outside (high lux), the LED doesn't need to be bright. The model should predict LOW values.

---

## 🔍 Root Cause Analysis

### The Issue: Synthetic Training Data

The current `tree_model.json` was trained on **synthetic data** generated by `train_brightness_model.py`.

The synthetic data generation logic has a flaw:

```python
# In the synthetic data generation:
if lux > 5000:
    intensity = np.random.uniform(0, 10) if motion else np.random.uniform(0, 5)
elif lux > 1000:
    intensity = np.random.uniform(10, 30) if motion else np.random.uniform(0, 15)
```

**Problem:** The synthetic data doesn't perfectly match real-world behavior because:
1. Random noise was added: `intensity += np.random.normal(0, 5)`
2. Edge cases might not be well-represented
3. The model learned from ~2,800 synthetic samples, not real user preferences

### What the Model Learned

Looking at your test predictions from training:
```
Morning + Bright + Motion (lux=7000) → 7.83%  ✓ Correct (low brightness)
```

But in your real system:
```
Lux > 2000 + Motion → 70% ❌ Wrong!
```

**This means:** Either:
1. Your `lux_norm` calculation is off (unlikely after our fix)
2. The `effective_need` feature is being calculated incorrectly
3. The model file is outdated or corrupted

## 🩺 Diagnostic Steps

### Step 1: Check Feature Calculation

Add this debug output to your ESP32 code to see what's being fed to the model:

```cpp
Serial.printf("DEBUG: lux=%.1f, lux_norm=%.4f, motion=%d, eff_need=%.4f\n",
              lux, lux_norm, motion, effective_need);
```

**Expected values when lux=2000, motion=1:**
```
lux_norm = 2000 / 14966 = 0.1336
effective_need = (1 - 0.1336) * 1 = 0.8664
```

If `effective_need` is high (>0.8), the model SHOULD predict high brightness. But that's wrong because we're in bright conditions!

### Step 2: The Real Problem - Model Logic Flaw

The `effective_need` feature has a conceptual flaw for your use case:

```
effective_need = (1 - lux_norm) × motion
```

**What it encodes:**
- High lux → low need → should predict LOW brightness ✓
- Low lux + motion → high need → should predict HIGH brightness ✓

**But the issue is:**
The model was trained on data where high `effective_need` → high brightness.
In your case at lux=2000 with motion:
- `lux_norm = 0.13` (low, because maxLux is 14,966)
- `effective_need = 0.87` (HIGH!)
- Model thinks: "High need → bright LED!" → predicts 70%

**The problem:** 2000 lux is already BRIGHT for indoor use, but it's only 13% of the max range, so `effective_need` is still high!

The best fix is to retrain the model with YOUR actual usage data.

**How to do it:**

1. **Collect real data for 1-2 weeks:**

3. **Retrain:**

   ```bash
   # Upload new model to ESP32
   ```

### Solution 2: **Adjust maxLux Value** (Quick Fix)

If 2000 lux feels "bright" to you, lower the `maxLux` so it becomes a higher percentage:

In `preprocessing.json`, change:
```json
{
  "max_lux": 5000.0     // Instead of 14966
}
```

Now when lux=2000:
- `lux_norm = 2000 / 5000 = 0.40` (40% of max)
- `effective_need = (1 - 0.40) * 1 = 0.60` (lower)
- Model predicts lower brightness

**To apply:**
1. Edit `preprocessing.json` in your project
2. Re-run `upload_to_spiffs.py`
3. Upload SPIFFS data to ESP32

### Solution 3: **Use Online Learning** (Long-term)

The model will adapt over time as you use it:

1. When you see lux=2000, motion=1, DT=70%:
   - Turn potentiometer to -50 (reduces to 20%)
   - System detects the adjustment
   - Collects training sample: `lux=2000, motion=1 → target=20%`

2. After 50 such adjustments:
   - Model retrains automatically
   - Learns your preferences
   - Future predictions improve

### Solution 4: **Add Rule-Based Override** (Immediate Fix)

Add a simple rule in the ESP32 code:

```cpp
// After ML prediction:
float dtBrightness = predictBrightness(...);

// Override for bright conditions
if (lux > 2000 && motion == 1) {
    dtBrightness = min(dtBrightness, 20.0f);  // Cap at 20%
}
if (lux > 5000) {
    dtBrightness = min(dtBrightness, 10.0f);  // Cap at 10%
}
```

## 📝 Code to Add (Rule-Based Override)

Add this after line where `dtBrightness` is calculated:

```cpp
// ---- ML inference ----
float dtBrightness = predictBrightness(lux_norm, motion, hour_sin,
                                       hour_cos, time_enc, effective_need);

// RULE-BASED OVERRIDE: Cap brightness in bright conditions
if (lux > 2000.0f && motion == 1) {
    float cappedValue = 20.0f;  // Max 20% when bright + motion
    if (dtBrightness > cappedValue) {
        Serial.printf("  [Override] Bright condition (%.0f lux) - capping DT %.1f%% -> %.1f%%\n", 
                      lux, dtBrightness, cappedValue);
        dtBrightness = cappedValue;
    }
}
if (lux > 5000.0f) {
    float cappedValue = 10.0f;  // Max 10% when very bright
    if (dtBrightness > cappedValue) {
        Serial.printf("  [Override] Very bright (%.0f lux) - capping DT %.1f%% -> %.1f%%\n", 
                      lux, dtBrightness, cappedValue);
        dtBrightness = cappedValue;
    }
}

lastDTPrediction = dtBrightness;
```
